1.3 System Constraints

Security Constraints
• Role-Based Access: Separate portals for Admin and Student to ensure sensitive operations (like result publication) are restricted.
• Authentication: Future implementation must include secure login to protect personal student details and exam records.
• Data Privacy: Encryption of student data is required to comply with privacy standards for educational records.
Scalability Constraints
• High Volume: System must efficiently process thousands of student registrations and exam applications simultaneously.
• Concurrent Access: Architecture must support simultaneous access from hundreds of schools and thousands of students during result cycles.
• Multi-Region: Scalable organizational hierarchy (Region > Centre > School) to manage growth across Maharashtra.
Database Constraints
• Relational Storage: All core entities (Students, Exams, Results) must be stored in an RDBMS to maintain complex relationships.
• Data Integrity: Strict consistency required between applications and results using unique applicationId mappings.
• JSON Support: Flexible storage for dynamic paper structures within strictly typed relational tables.

2.4 Entity Relationship (ER) Diagram

The following diagram illustrates the relational mapping and cardinality between the backend entities.

Figure 1 — Entity Relationship (ER) Diagram (High-Fidelity Architectural Overview)

Relational Mapping & Cardinality Details

The system utilizes a Relational Database Management System (RDBMS) with the following mapping strategies:

  • One-to-Many (1:N) Bidirectional:
    Region ↔ Exam Centre ↔ School ↔ Student. Each parent node contains a collection of children, while each child maintains a reference (Foreign Key) to its parent.
  • Many-to-One (N:1):
    Multiple Students and Exams map to a single Application instance through Foreign Keys (student_id, exam_no) in the ExamApplication entity.
  • One-to-One (1:1):
    Each ExamApplication maps to exactly one ExamResult, ensured by a UNIQUE constraint on the application_id foreign key.
  • Many-to-Many (N:M) via Bridge:
    The relationship between Student and Exam is Many-to-Many, realized through the ExamApplication bridge entity to maintain relational integrity.
  • Lazy Loading:
    Mappings are configured for Lazy Fetching to optimize performance when retrieving large datasets like Student or School lists.

2.5 Functional Data Flow & Lifecycle

This section illustrates the dynamic movement of data through the system, from initial user registration to final result publication.

Figure 2 — End-to-End Operational Lifecycle: From Registration to Offline Examination & Result Publication

Hybrid Lifecycle Workflow Analysis

1. Digital Setup:

Students register and apply for exams through the Portal. Management configures exam schedules and generates secure Digital Hall Tickets for students.

2. Offline Execution:

Examinations are conducted Physically (Offline) using OMR/Paper sheets at designated centers. Hall tickets are verified against the digital record during entry.

3. Manual Evaluation:

Physical answer scripts are collected and Evaluated Manually by checkers. Marks are then digitized and uploaded into the Central Database through the Admin Management interface.

4. Result Publication:

Once marks are finalized, the system Automatically Generates Results, publishes them to the student dashboards, and provides performance analytics.

Proposed Future Enhancements (PROPOSED ALGORITHMS)

To further automate the Maharashtra Rashtrabhasha Board's operations and enhance system integrity, the following technical algorithms are proposed for implementation in Phase 2.

4.1 Composite Seat Number Allocation Algorithm

A deterministic algorithm to generate unique, traceable seat numbers for every student appearing in an exam cycle. Instead of random IDs, the seat number is a 12-digit composite key designed for administrative routing.

Logic: [RR][CCC][SSS][IIII]
• RR (Region Code): 2 digits (e.g., 01 for Pune, 02 for Mumbai).
• CCC (Centre Code): 3 digits identifying the specific exam centre.
• SSS (School Code): 3 digits for the student's base school.
• IIII (Index): 4-digit auto-incrementing serial number within that school.

Benefit: Allows instantaneous sorting of answer sheets at the board office without querying the database, as the physical destination is encoded in the ID.

4.2 Statistical Anomaly & Fraud Detection

A pattern-matching algorithm to identify suspicious result clusters. It calculates the Standard Deviation (σ) of marks within each exam centre.

  • Identical Pattern Check: Flags centres where >30% of students have identical marks in paper-wise breakdowns.
  • Outlier Detection: Flags centres whose AvgScore > (GlobalAvg + 3σ).
  • Passing Rate Spike: Triggers alert if a centre's passing rate increases by >40% compared to previous 3 years.

4.3 Verifiable Marksheet Hashing

To eliminate forgery, every marksheet will carry a digital fingerprint generated via the SHA-256 Hashing algorithm.

hash = SHA256(studentId + examNo + totalMarks + secretKey)

The hash is embedded in a QR code. External verifiers (colleges/employers) scan it to hit a /verify endpoint that re-calculates the hash and confirms authenticity.

4.4 Intelligent Conflict-Free Scheduling

A Graph Coloring Algorithm based approach to manage thousands of papers across hundreds of centers with zero conflicts.

Mechanism: The system identifies "Student Overlap" — cases where one student appears for multiple exams (e.g., Prathamik and Prabodh). These papers are treated as connected nodes in a graph. The algorithm assigns the minimum number of "time-colors" such that no two connected nodes share the same time slot, optimizing the total exam duration for the Board.

5.3 N+1 Query Problem Resolution

During initial load testing of the /getAllApplications endpoint, a classic ORM N+1 query problem was discovered. Fetching a list of applications inadvertently triggered separate database queries to fetch the associated Student and Exam entities for every single application in the list.

This resulted in massive overhead (e.g., 100 applications = 1 query for apps + 100 queries for students + 100 queries for exams = 201 total queries), leading to response times of over 6 seconds.

The Solution: The backend was refactored to implement rigorous Data Transfer Objects (DTOs) combined with custom JPQL JOIN FETCH queries. Instead of returning deeply nested Hibernate proxy entities that trigger lazy loading, the API now projects exactly the required fields into a flat ApplicationDTO.

// Example of the flat, efficient DTO output (output_json/GET_getAllApplications.json) { "applicationId": 1, "examNo": 5, "examName": "Rashtrabhasha Prabodh Pariksha (May 2026)", "studentId": 1, "studentName": "Aarav Sharma", "status": "PENDING" }
Conclusion: Implementing caching using Redis/Spring Cache resulted in up to ~268x faster response times for heavy payloads like applications and results. Furthermore, the integration of DTOs permanently resolved the N+1 query vulnerability, ensuring reliable list endpoint delivery in ~20-40ms, down from 1-6 seconds.
